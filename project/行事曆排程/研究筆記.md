# AI行事曆排程系統 研究筆記

## 背景與動機

目前 daily-sync 系列 skill（sub-daily-check + sub-apple-sync）已經可以把 `管理行程/` 資料夾裡的截圖、課表整理進 `tasks.json`，再同步到 Calendar/Reminders/Notes。但每次跑都是讓 LLM 重新讀圖、重新判斷一次，token 消耗偏高。這份筆記研究：能不能用「比對邏輯」取代一部分 LLM 判斷，並搭配可視化介面統一檢視、確認後才派發。

現階段只做研究/架構設計，不寫程式。

## 目標

1. **降低 token 消耗**：能用決定性程式比對解決的部分，不要每次都丟給 LLM 重新判斷。
2. **保留彈性**：比對不到、需要語意理解的部分，還是交給 LLM。
3. **可視化＋人工確認**：在 VSCode 裡有一個介面可以統一檢視/修改排程狀態，確認後才派發到 Calendar/Reminders/Notes——不是自動全信任派發。

## 核心構想：比對邏輯（甘特圖式思維）

類比甘特圖——把「已知/固定」的行程當作基準軸，新進來的資料只需要「比對」有沒有對上，而不是每次重新理解整份資料。有兩個比對方向，**目前都要考慮，尚未決定選哪個或並行**：

### 方向 A：新資料 vs 既有行事曆事件

- sub-daily-check 讀到的截圖/課表資料，跟 Calendar.app 裡已經存在的事件做比對。
- 比對到的：代表這件事已經同步過，不需要再讓 LLM 判斷一次，直接靠 hash/去重機制跳過。
- 比對到但有差異的（時間改了、多了附件等）：用決定性程式更新對應欄位，不用整段重新丟給 LLM。
- 沒比對到的：才需要 LLM 語意判斷是不是新行程。
- 現有 sub-apple-sync 的 `.apple-sync-state.json` hash 去重已經是簡化版比對邏輯（有沒有存在），可以在這基礎上擴充「差異比對」（存在但內容變了要不要更新）。

### 方向 B：tasks.json vs 標準行程樣板

- 把每天/每週固定會發生的行程（上課時間、固定會議）存成「樣板」。
- 新的 tasks.json 進來時，先跟樣板比對哪些是重複發生的固定行程，直接套用樣板欄位（含常見附件），不用每次重新判斷。
- 只有樣板比對不到的「例外行程」才需要 LLM 從頭判斷。
- 待研究：樣板怎麼定義、怎麼處理樣板本身的更新（例如課表換學期）。

## 附件比對

使用者提到「比對到的行程的附件也可以排進來」——初步構想：

- 附件來源可能是 `管理行程/` 資料夾裡跟該行程相關的檔案（截圖、報名表等）。
- 比對邏輯除了判斷「這是不是同一件事」，還要判斷「這個事件過去/樣板有沒有慣用附件」，有的話一併帶入。
- 待研究：附件怎麼存進 Calendar 事件——AppleScript 能不能夾帶檔案，還是只能存路徑/連結。

## 可視化介面：VSCode 面板

使用者偏好在 **VSCode 裡有統一介面處理專案狀態，確認後才派發出去**，不是先做一個獨立的靜態 HTML 甘特圖。

- 參考 `DocOutline/` 的模式（TypeScript + VSCode Extension API + Webview），但這次面板不是「插入文字到 .tex」，而是：
  - 顯示目前比對結果（哪些行程比對到既有事件/樣板、哪些是全新的、哪些有差異待確認）。
  - 讓使用者在面板上確認/修改後，才觸發派發（呼叫 sub-apple-sync 的邏輯寫入 Calendar/Reminders/Notes）。
  - 定位比較接近「審核佇列」UI，不是畫甘特圖本身，但畫面呈現可以參考時間軸排列方式。
- 待研究：Webview 要不要引入前端框架，還是延用 DocOutline 目前純 HTML/JS 的極簡原則（`延伸專案/技術棧.md` 目前建議沒必要時不用框架）。

## 開放問題（尚待研究決定）

1. 比對邏輯要選方向 A、方向 B，還是兩者並行？各自的比對 key 怎麼定義（時間＋標題？地點？）
2. 樣板（方向 B）怎麼儲存、怎麼更新，會不會反而增加維護成本？
3. 附件怎麼跟 Calendar 事件關聯，AppleScript 的能力邊界在哪？
4. VSCode 面板要即時讀 apple-sync 的 state 檔，還是需要另外定義一份給面板用的比對結果格式？
5. 這個系統最終要取代 daily-sync 的哪些步驟，還是疊加在既有流程之上？
