# 技術棧總覽

這份文件整理 Life.os 裡「實際會執行程式碼」的部分用了什麼技術，給日後寫新小工具、或想改既有程式時當參考索引。細節仍以各專案自己的技術文件為準（`每日同步技術文件/ReadMe.md`、`DocOutline/ReadMe.md`）；這裡只做跨專案的整理與選型建議。

目前這個 repo 裡有三種不同性質的技術堆疊，對應三種不同的工具形態：

## 1. Claude Code Skill 自動化（daily-check / ios-sync / notes-sync / daily-sync）

用途：資料整理 + 跟 macOS 系統 App（Calendar/Reminders/Notes）互動的自動化。

- **語言**：Node.js，只用內建模組（`fs`、`path`、`crypto`、`child_process`），**沒有任何 npm 套件**、沒有 `package.json`、沒有 build step——腳本用 `node xxx.js` 直接執行。
- **系統整合**：`child_process.execFileSync('osascript', ['-e', script])` 呼叫 AppleScript，操作 Calendar.app / Reminders.app / Notes.app。
- **架構模式**：兩層分工——
  - LLM 判斷層：`SKILL.md`，負責需要語意理解的部分（看懂截圖課表、判斷檔案是否過期），輸出結構化 JSON（`tasks.json`）當資料契約。
  - 決定性程式層：`scripts/*.js`，純粹消費 JSON、做 dedup / 建立 / 更新 / 刪除，不依賴 LLM 重新判斷，行為可重現。
- **去重機制**：`crypto.createHash('sha1')`，對「任務內容」算 hash 存在本機 state 檔（`.ios-sync-state.json` / `.notes-sync-state.json`，不進 git），重跑不會產生重複資料。
- **權限模型**：`.claude/settings.json` 的 `permissions.allow` 白名單刻意收斂到這幾個腳本實際會用到的 `Read`/`Write`/`Bash` 範圍，不開放整個 repo 的寫入權限。
- **沒有**：測試框架、lint、CI——單機個人工具，用真實跑過（建立/改時間/重跑/刪除）驗證正確性。

適合場景：**任何「把某個資料夾的資料整理好、同步進 macOS 原生 App」的需求**，不需要引入任何套件，Node 內建 + AppleScript 就夠。

## 2. VSCode Extension（DocOutline）

用途：需要跟 VSCode 編輯器本身互動（讀取/插入文字、做面板 UI）的工具。

- **語言**：TypeScript `^5.5.4`，`strict: true`，編譯目標 ES2022 / Node16 module resolution（見 `tsconfig.json`）。
- **平台**：VSCode Extension API，`engines.vscode: ^1.90.0`。
- **核心第三方套件**：`pdfjs-dist ^4.6.82`——用來 parse PDF 的 outline（書籤）、annotations（highlight 標註）、text content。動態 `import()`（ESM）載入，見 `pdf/pdfDocumentCache.ts`。
- **devDependencies**：`@types/node`、`@types/vscode`、`typescript`——沒有額外的 lint/format/test 工具鏈（MVP 階段刻意精簡）。
- **外部系統整合**：Skim.app（可腳本化的 macOS PDF 閱讀器）+ AppleScript（`osascript`），把 Skim 的螢光筆標註「以嵌入 PDF 附註」的方式存檔，因為 Skim 預設用 extended attributes 存標註，一般存檔不會寫進 PDF 本身。
- **UI**：Webview，內嵌 HTML/CSS/JS（`webview/panel.ts`），沒有用 React/Vue 等前端框架——面板邏輯簡單，沒必要引入框架。
- **Build**：`tsc -p ./`（`npm run compile`）、`tsc -w -p ./`（`npm run watch`，`F5` 啟動 Extension Development Host 時自動跑）。

適合場景：**需要讀取/操作使用者正在編輯的檔案、或需要一個互動面板 UI** 的工具。比 Claude Code skill 重（要寫 TypeScript、要處理 VSCode API 生命週期),但換來的是能嵌進編輯器工作流程本身。

## 3. LaTeX 排版（聖經陪讀教材）

用途：中文技術/知識文件的排版產出。

- **引擎**：`xelatex`（處理中文必須用這個，不能用預設 `pdflatex`）。
- **套件**：`ctex`（中文排版支援）。
- **本機安裝**：`brew install --cask basictex`（精簡版，再 `sudo tlmgr install ctex xecjk` 補中文套件）或 `brew install --cask mactex`（完整版，~5GB，免另外裝套件）。
- **VSCode 整合**：LaTeX Workshop 擴充套件（`James-Yu.latex-workshop`），設定 `latex-workshop.latex.tools`/`recipes` 指向 `xelatex`，開 `autoBuild.run: onSave` 就能存檔即時看 PDF 預覽。
- **前導碼慣例**：見 `CLAUDE.md` 的「慣例」章節（`\documentclass{article}` + `\usepackage{ctex}`）。

## 日後寫新小工具時怎麼選

| 需求 | 參考模式 | 理由 |
|---|---|---|
| 整理資料、跟 macOS 原生 App（Calendar/Reminders/Notes/Mail...）互動 | daily-sync 模式（Node 內建 + AppleScript） | 不需要任何套件，AppleScript 是跟 macOS App 互動最直接的方式 |
| 需要在 VSCode 裡讀取/插入編輯器內容、做互動面板 | DocOutline 模式（TypeScript + VSCode Extension API） | 只有 Extension API 能拿到「目前聚焦的編輯器」與游標位置 |
| 只是分類/判斷/摘要這類語意任務，沒有需要程式化的副作用 | 純 Claude Code skill（`SKILL.md`），不寫程式 | 語意判斷交給 LLM 本身最省事，不需要額外维护程式碼 |
| 要 parse PDF 內容（大綱/標註/文字） | `pdfjs-dist`（DocOutline 已驗證可用） | 已經在這個 repo 用過，遇到坑（見 `DocOutline/ReadMe.md`）都記錄過了 |
