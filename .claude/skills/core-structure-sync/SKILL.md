你現在要檢查 Life.os 的資料夾結構有沒有跟文件（`README.md`、`CLAUDE.md`）脫節。這是會被重複執行的檢查，請照下面步驟走，不要跳過「詢問使用者」這一步。

## 1. 取得目前實際結構

列出目前資料夾結構（第一層 + 有意義的第二層即可，不用整棵樹），排除 `.git`、`node_modules`、`.DS_Store`、build 產物（如 `out/`）等雜訊：

```
find . -maxdepth 2 -type d \
  -not -path './.git*' -not -path '*/node_modules*' -not -path '*/.git' \
  | sort
```

## 2. 讀文件裡記錄的結構

讀 `CLAUDE.md` 的「資料夾結構」章節，以及根目錄 `README.md` 的「依場景查」「幾條線」章節，列出目前這兩份文件提到的資料夾。

## 3. 比對，列出差異

三種差異都要抓：

- **新增**：實際存在、但兩份文件都沒提到的資料夾。
- **消失/改名**：文件提到、但實際已經不存在的資料夾——特別注意用途看起來相同、只是名稱不同的（例如英文資料夾消失、同時出現對應的中文資料夾），這種通常是改名/重組，不是單純刪除，判斷時要指出「看起來像 A 改名成 B」。
- **內容漂移**：資料夾還在，但裡面的用途看起來已經跟文件描述的不一樣了（例如原本說「目前為空」但已經有實質內容）。

如果沒有任何差異：跟使用者說「資料夾結構跟文件一致，不需要更新」，結束，不要往下做。

## 4. 詢問使用者原因（不可跳過）

**發現差異後，不要自己腦補原因就動手改文件。** 用一句話列出每一項偵測到的變動，然後問使用者：

- 這是新增了什麼功能/用途？還是單純改名、搬移整理？
- 如果是新功能，簡短說一下是做什麼用的（要寫進 README 的場景清單）。

等使用者回覆之後才能繼續下一步。這一步的目的是避免「改了 A，但外面看不出來多了什麼」——沒有使用者說明的「為什麼」，文件更新只是抄結構，抓不到重點。

## 5. 依使用者的說明更新文件

- **CLAUDE.md**：更新「資料夾結構」章節，反映目前實際路徑（改名就直接改路徑；新增就照使用者說的用途加一行）。
- **README.md**：
  - 新功能 → 在「依場景查」對應場景加一條（或新增一個場景），用使用者剛剛講的話改寫成一句「我想 ___」的場景描述。
  - 純改名/搬移 → 只更新既有條目的路徑，不要新增重複場景。
  - 如果是全新的一條「線」（不屬於現有分類），也同步更新「專案裡的幾條線」章節。

兩份文件的「資料夾結構/場景」內容不能只改一份、留另一份繼續過期。

## 6. 回報

用 2-3 句話說明：偵測到什麼變動、使用者說的原因是什麼、改了 README.md 和/或 CLAUDE.md 的哪個部分。不要整段貼文件 diff。
