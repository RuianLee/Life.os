你現在要執行 Life.os 的「每日行程檢查」。這是一個會被重複執行（手動或排程）的任務，請嚴格照下面步驟做，不要自由發揮格式。

## 0. 準備

- 取得今天的日期（`date +%F`），以下都用 `YYYY-MM-DD` 表示今天。
- 讀取 `schedule/schedule-log.md`（若不存在視為空檔案），了解之前執行過的紀錄，避免重複判斷同一批已經處理過的檔案。
- 讀取 `second-brain/主題統整.md`，看有沒有「目前每日計劃」表格（固定/長期的每日例行事項）。

### 0a. 讀取 schedule/（先萃取、再讀取，省 token）

`schedule/` 每個分類子資料夾（例如 `休開/`、`2026召會生活/`）底下可能有一個 `原始檔/` 子資料夾，用來存放「已經轉譯過的原始截圖/PDF」。規則：

1. **先看分類資料夾最上層（不含 `原始檔/`）**：
   - 如果是 `.md`／純文字檔：直接 Read 讀文字內容即可，不用再看任何圖片——這就是已經萃取好的版本，省下大量圖片 token。
   - 如果是圖片／PDF（代表這是還沒處理過的新檔案，可能是使用者剛加進來的，也可能是還沒補萃取的舊檔）：用 Read 工具完整讀取內容，**轉譯成同名、副檔名改 `.md` 的純文字/Markdown 檔案**，放在同一層（`原始檔/` 外面）。轉譯時要保留檔案裡的**全部**資訊，不要只挑「今天～未來7天」這個窗口內的部分——因為這份 `.md` 之後會被重複使用很多天，之後每次執行只需要讀這份 `.md`，不用再用視覺方式讀原始截圖。
   - 轉譯完成後，把原始圖片/PDF **搬進**同一層底下的 `原始檔/` 子資料夾（不存在就 `mkdir -p` 建立），這樣它就不會被下次執行誤判成「還沒處理的新檔案」。如果 `原始檔/` 底下已經有同名檔案（代表這是同一份文件的內容更新版），搬移前幫舊檔加上日期後綴（例如 `原檔名_2026-07-05.jpg`）避免覆蓋，保留歷史版本。
2. **`原始檔/` 底下的檔案本身不用讀**，它們只是萃取後的存檔備查，已經沒有資訊價值（該有的都在同層的 `.md` 裡了）。
3. **重要：`.md` 是新增出來的，不是取代 source_file 的寫法**。之後產生 `tasks.json` 時，`source_file` 欄位仍然要寫「概念上的原始檔案」路徑（例如仍寫 `schedule/休開/休訓讀經進度.jpg`，即使實際檔案已經搬進 `原始檔/` 或改了名字），不要因為多了 `.md` 版本或原始檔案改名/搬移，就換一個新的 `source_file` 寫法——`id` 是用 `source_file + date + title` 算出來的，換掉會讓既有任務被誤判成新任務，造成 Apple 端重複建立。每份 `.md` 的開頭也會記錄它對應的「概念上原始檔案路徑」，遇到改名/搬移的情況直接照 `.md` 裡寫的沿用即可，不用自己重新判斷。

## 1. 整理今天/明天/本週待辦（tasks.json，給 sub-apple-sync 用）

- 找出「今天」「明天」「未來 7 天內」有明確日期、需要完成的事項。這個 7 天窗口只影響 `tasks.json`（機器讀，供 Apple 同步預先建立未來幾天的 Calendar/Reminders），跟下一節的人讀檔案窗口是兩回事，不要混在一起。
- 輸出結構化版本到 `02-zettelkasten/03-Calendar/<today>/tasks.json`（若資料夾不存在就建立），是一個 JSON array，每個任務物件的欄位：
  ```json
  {
    "id": "<sha1(source_file + date + title) 取前 12 碼 hex>",
    "title": "簡短標題，例如 16:00 後至三會所到寢室",
    "date": "YYYY-MM-DD",
    "time": "HH:MM 或 null（沒有明確時間就是 null）",
    "type": "event 或 reminder（有明確時間用 event，只有日期用 reminder）",
    "source_file": "相對於 repo 根目錄的來源檔案路徑",
    "notes": "補充說明，可為空字串"
  }
  ```
  **關鍵限制**：計算 `id` 時絕對不可以把「今天執行的日期」算進去，只能用「任務本身的 date + title + source_file」。這樣同一件事不管哪一天被重新產生，id 都必須相同 —— 這是後續 `sub-apple-sync` 判斷是否重複建立事件的唯一依據。用 shell 算：`echo -n "<source_file>|<date>|<title>" | shasum | cut -c1-12`。
  - 只收錄「今天／明天／未來 7 天」範圍內的任務，不要把整個 schedule 的所有日期都塞進去。

### 1a. 固定每日計劃（來自 主題統整.md）

- 如果 `second-brain/主題統整.md` 有「目前每日計劃」表格，把表格每一列都轉成**今天**的一筆 task：
  - `title`：該列「內容」欄位＋時長，例如「生活紀錄、生產力養成（35 分鐘 * 1）」。
  - `date`：今天。
  - `time`：`null`（表格只有時長沒有明確時鐘時間，一律當 `reminder`，除非使用者在表格裡有另外註明具體時段）。
  - `type`：`"reminder"`。
  - `source_file`：`"second-brain/主題統整.md"`。
  - `id` 一樣用 `sha1(source_file + date + title)`：因為 `date` 每天不同，id 每天自然不同，`sub-apple-sync` 既有的「反向清理」邏輯就會自動把前一天沒出現的舊 reminder 標記完成——不需要另外做狀態追蹤或判斷「今天有沒有設定過」，表格本身就是唯一的開關（表格拿掉，這步就自然不再產生任何項目）。
- 這些項目跟第 1 節找到的一次性待辦，一起合併寫進同一份 `02-zettelkasten/03-Calendar/<today>/tasks.json`，不用分開檔案。
- 如果 `主題統整.md` 沒有這個表格，或表格是空的，跳過這步，不用報錯或提醒使用者。

## 2. 產生人讀檔案

不要把「今天」跟「未來一週/一個月/一年」混在同一份檔案裡——分成兩份，各自負責不同的時間尺度：

### 2a. `02-zettelkasten/03-Calendar/<today>/<today>.md`：只放「今天」

- 只放**今天**的待辦（不要放明天、也不要放未來 7 天），格式維持 checkbox 條列、附關鍵時間點。
- 開頭簡短註明資料來源（哪些檔案），不用像以前一樣寫一大段「涵蓋範圍限制」，並且幫我以列點的形式寫出來——那些交給 2b 的月計劃/週計劃區塊處理。
- 如果想看更遠的規劃，直接參考 `schedule/計劃總覽.md`。

### 2b. `schedule/計劃總覽.md`：年/月/週的滾動總覽

這份檔案**不是每天新建一份、也不放在 02-zettelkasten/03-Calendar 底下**——它常駐在 `schedule/`，每次執行 sub-daily-check 都覆寫更新同一份，反映「現在」的規劃全貌。分三段：

1. **年計劃**：這一年（依 `second-brain/主題統整.md` 的「短期（年內）」清單等長期目標）大方向要做什麼，變動很慢，通常整段照抄複製過去、有異動才更新。
2. **月計劃**：至少涵蓋「今天所在月份＋接下來 2 個月」共 3 個月，每個月一小節，用條列列出該月已知的重點（不用逐日列，那是週計劃的工作）。沒有資料的月份就寫「尚無資料」，不用省略那個月的標題（讓使用者一眼看出規劃還沒排到那裡）。
3. **週計劃**：今天～未來 7 天，依日期分節、checkbox 條列——這就是原本 daily-plan md 裡「今天/明天/未來7天」那部分的內容，現在搬來這裡。這段內容的資料範圍跟 `tasks.json` 一致，可以直接重用同一次判斷的結果，不用重新分析一次。

## 3. 判斷是否搬移過期檔案（保守原則）

對 `schedule/` 底下每個檔案，判斷：

1. 檔案內提到的所有日期是否都已經早於今天？
2. 內容是否明顯是「一次性事件的紀錄」（例如某次活動的課表、名冊、報名截止公告），而不是「可重複使用的範本/清單」（例如裝備檢核表這種下次還可能照抄使用的通用清單）？

**只有兩個條件都成立，才把檔案搬到 trash-can/**，搬移時保留原始相對路徑，例如：
```
mv "schedule/青職短訓/短訓課程表.jpg" "trash-can/schedule/青職短訓/短訓課程表.jpg"
```
（`trash-can/<原始相對路徑>`，資料夾不存在就先 `mkdir -p`）

如果不確定（例如日期過期但看起來像可重複使用的範本），**不要搬移**，改成在 log 裡記一筆「建議清理但保留」。寧可少搬、不要搬錯。

## 4. 寫 log

把這次執行的結果 append 到 `schedule/schedule-log.md`（不可覆寫既有內容），格式：

```markdown
## <today> daily-check

- 已搬移：`<相對於 schedule/ 的路徑>` → `trash-can/schedule/<路徑>`
  原因：<一句話原因>
- 建議清理但保留：`<相對於 schedule/ 的路徑>`
  原因：<一句話原因>
```

如果這次沒有任何搬移或建議，也要寫一段最簡短的紀錄（例如「本次無檔案需要處理」），保持每次執行都有留痕。

## 5. 回報

執行完後用 2-3 句話跟使用者說明：整理了哪些待辦（含固定每日計劃有沒有帶入）、搬了幾個檔案（附原因）、有沒有「建議清理但保留」的項目。不要整段貼 tasks.json 或 log 的原始內容。
