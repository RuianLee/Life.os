你現在要執行 Life.os 的「每日行程檢查」。這是一個會被重複執行（手動或排程）的任務，請嚴格照下面步驟做，不要自由發揮格式。

## 0. 準備

- 取得今天的日期（`date +%F`），以下都用 `YYYY-MM-DD` 表示今天。
- 讀取 `管理行程/` 底下所有檔案（含子資料夾）。圖片用 Read 工具直接看內容（行事曆、課表、名冊等常常是截圖）。
- 讀取 `管理行程/schedule-log.md`（若不存在視為空檔案），了解之前執行過的紀錄，避免重複判斷同一批已經處理過的檔案。

## 1. 整理今天/明天/本週待辦

- 找出「今天」「明天」「未來 7 天內」有明確日期、需要完成的事項。
- 寫一份人讀的整理到 `每日重點整理/<today>/<today>.md`（若資料夾不存在就建立）。格式參考既有的 `每日重點整理/2026-07-05/2026-07-05.md`：依日期分節、用 checkbox 條列、附上關鍵時間點。開頭註明資料來源與資料涵蓋範圍的限制（例如「管理行程 資料夾只排到 X 月 X 日」）。
- 同時輸出結構化版本到 `每日重點整理/<today>/tasks.json`，是一個 JSON array，每個任務物件的欄位：
  ```json
  {
    "id": "<sha1(source_file + date + title) 取前 12 碼 hex>",
    "title": "簡短標題，例如 16:00 後至三會所到寢室",
    "date": "YYYY-MM-DD",
    "time": "HH:MM 或 null（沒有明確時間就是 null）",
    "type": "event 或 reminder（有明確時間用 event，只有日期用 reminder）",
    "source_file": "相對於 repo 根目錄的來源檔案路徑",
    "notes": "補充說明，可為空字串"
  }
  ```
  **關鍵限制**：計算 `id` 時絕對不可以把「今天執行的日期」算進去，只能用「任務本身的 date + title + source_file」。這樣同一件事不管哪一天被重新產生，id 都必須相同 —— 這是後續 `sub-apple-sync` 判斷是否重複建立事件的唯一依據。用 shell 算：`echo -n "<source_file>|<date>|<title>" | shasum | cut -c1-12`。
  - 只收錄「今天／明天／未來 7 天」範圍內的任務，不要把整個 管理行程 的所有日期都塞進去。

## 2. 判斷是否搬移過期檔案（保守原則）

對 `管理行程/` 底下每個檔案，判斷：

1. 檔案內提到的所有日期是否都已經早於今天？
2. 內容是否明顯是「一次性事件的紀錄」（例如某次活動的課表、名冊、報名截止公告），而不是「可重複使用的範本/清單」（例如裝備檢核表這種下次還可能照抄使用的通用清單）？

**只有兩個條件都成立，才把檔案搬到 垃圾桶/**，搬移時保留原始相對路徑，例如：
```
mv "管理行程/青職短訓/短訓課程表.jpg" "垃圾桶/管理行程/青職短訓/短訓課程表.jpg"
```
（`垃圾桶/<原始相對路徑>`，資料夾不存在就先 `mkdir -p`）

如果不確定（例如日期過期但看起來像可重複使用的範本），**不要搬移**，改成在 log 裡記一筆「建議清理但保留」。寧可少搬、不要搬錯。

## 3. 寫 log

把這次執行的結果 append 到 `管理行程/schedule-log.md`（不可覆寫既有內容），格式：

```markdown
## <today> daily-check

- 已搬移：`<相對於 管理行程/ 的路徑>` → `垃圾桶/管理行程/<路徑>`
  原因：<一句話原因>
- 建議清理但保留：`<相對於 管理行程/ 的路徑>`
  原因：<一句話原因>
```

如果這次沒有任何搬移或建議，也要寫一段最簡短的紀錄（例如「本次無檔案需要處理」），保持每次執行都有留痕。

## 4. 回報

執行完後用 2-3 句話跟使用者說明：整理了哪些待辦、搬了幾個檔案（附原因）、有沒有「建議清理但保留」的項目。不要整段貼 tasks.json 或 log 的原始內容。
